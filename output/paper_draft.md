# 도구 사용형 AI 에이전트의 개인정보 과도 접근 측정: Exposure Area 기반 실증 연구

## 초록

도구(tool) 사용 능력을 갖춘 AI 에이전트가 협업 업무를 수행할 때, 개인정보를 얼마나 과도하게 접근하는지 정량적으로 측정하는 프레임워크를 제안한다. 본 연구에서는 가상 메일·캘린더·주소록 환경에서 "김민수와 회의 일정 조율" 업무를 부여하고, 무제한 데이터 접근(조건 A)과 최소정보 제한 접근(조건 B)을 비교하였다. 측정 지표로는 **Exposure Area** (Σ 민감도 × 관련성)를 새로이 정의하였다. 실험 결과, 조건 B는 조건 A 대비 접근 항목을 52개에서 5개로 90.4% 감소시켰으며, Exposure Area는 3.06에서 1.99로 35.0% 감소하였다. 특히 무관 항목 비율은 86.5%에서 0%로 완전히 제거되었고, 프롬프트 인젝션 악성 이메일 사고는 조건 A에서만 발생하였다. 업무 성공률은 두 조건 모두 100%로, 최소정보 접근이 업무 완료를 저해하지 않으면서 프라이버시 리스크를 근본적으로 낮춤을 확인하였다. 본 연구는 AI 에이전트의 프라이버시 보호 설계에 대한 실증적 기준을 제공한다.

---

## 1. 서론

AI 에이전트가 사용자의 메신저, 이메일, 캘린더, 주소록 등 민감한 개인 데이터에 접근하며 자율적으로 업무를 수행하는 사례가 급증하고 있다[1]. 이 과정에서 에이전트가 업무에 필요한 최소 정보만이 아니라, 불필요한 개인정보를 과도하게 열람할 가능성이 제기된다[2]. 또한 악의적으로 조작된 프롬프트 인젝션 이메일을 에이전트가 수신할 경우, 내부 연락처·이메일 내용을 외부로 유출하는 2차 사고로 이어질 수 있다.

기존 연구는 주로 "AI가 생성하는 응답의 개인정보 노출"에 초점을 두었으나[3,4], **에이전트가 도구 호출 과정에서 스스로 어떤 데이터를 읽는지** 측정하는 연구는 부족한 실정이다.

본 연구의 목적은 다음과 같다:
- 도구 사용형 AI 에이전트의 개인정보 접근 패턴을 정량적으로 측정할 수 있는 지표 **Exposure Area**를 제안한다.
- 무제한 데이터 접근 환경과 최소정보 필터링 환경을 비교하여, 프라이버시 보호와 업무 성공률의 트레이드오프를 실증한다.
- 프롬프트 인젝션 공격에 대한 에이전트의 노출 차단 효과를 실험적으로 검증한다.

---

## 2. 관련 연구

### 2.1 LLM 기반 AI 에이전트와 도구 접근

최근 GPT-4o, Claude 3.5 Sonnet 등은 tool use API를 통해 외부 시스템의 데이터를 읽고, 수정하고, 전송할 수 있다[5].但这种 능력은 "과도한 권한(over-privilege)" 문제를 야기한다 — 사용자가 요청한 작업 외의 데이터까지 접근·활용할 수 있기 때문이다.

### 2.2 개인정보 프라이버시 측정

GDPR, 개인정보보호법 등에서는 "최소정보 접근 원칙"을 요구하나[6], AI 에이전트 맥락에서의 정량적 측정 지표는 드물다. 본 연구의 Exposure Area는 민감도·관련성 곱을 합산하는 방식으로, 단순 항목 수가 아닌 **민감 정보의 함량**까지 고려한다는 점에서 기존 연구와 차별화된다.

### 2.3 프롬프트 인젝션 공격

Prompt injection은 AI 시스템의 지시를 우회하여 악의적 행위를 유도하는 공격으로[7], 특히 도구 접근이 가능한 에이전트에서는 데이터 유출이라는 심각한 결과를 초래할 수 있다. 본 연구에서는 악성 이메일에 삽입된 인젝션 페이로드가 에이전트의 접근 권한에 따라 어떻게 다른 결과를 초래하는지 실험하였다.

## 3. 방법론

### 3.1 연구 설계

본 연구는 시뮬레이터 기반 비교 실험으로, 동일한 업무 지시를 받은 Agent가 두 가지 접근 조건에서 어떻게 행동하는지 비교한다.

| 조건 | 데이터 제공 범위 | Agent 권한 | 목적 |
| --- | --- | --- | --- |
| **A (무제한)** | 연락처 15명 + 이메일 30건 + 캘린더 7일 (총 52개 항목) | `read_all` | 현행 시스템의 과도 접근 baseline |
| **B (최소정보)** | 김민수 연락처 1개 + 관련 이메일 2건 + 월/화 캘린더 2개 (총 5개 항목) | `read_filtered(task_type)` | 최소정보 제어 효과 검증 |

### 3.2 데이터셋

가상 데이터셋은 15명의 연락처, 30건의 이메일, 7일분의 캘린더 일정으로 구성하였다. 각 항목에는 민감도 분류를 위해 다음 태그를 부여하였다:
- **PII Level 1** (이름, 이메일, 전화): 업무 필수
- **PII Level 2** (부서, 직급): 업무 보조
- **PII Level 3** (건강·가족·여행·외근 등): 비업무

악성 이메일(e17)은 `unknown@prompt-hack.net` 발신자로, 프롬프트 인젝션 지시를 포함하도록 구성하였다.

### 3.3 Exposure Area 지표

단일 항목 \(i\)의 프라이버시 리스크는 다음과 같이 정의한다:

\[
EA_i = sens_i \times rel_i
\]

- **민감도 \(sens_i\)**: 항목이 포함하는 개인정보의 민감한 정도 (0~1)
- **관련성 \(rel_i\)**: 현재 업무와의 관련 정도 (0=무관, 1=완전 관련)

전체 Exposure Area는 다음과 같다:

\[
EA = \sum_{i=1}^{N} sens_i \times rel_i
\]

### 3.4 평가지표

| 지표 | 정의 |
| --- | --- |
| Exposure Area | 개인정보 노출 면적 |
| 평균 민감도 | 전체 접근 항목의 평균 민감도 |
| 무관 항목 비율 | 관련성 < 0.3 인 항목의 비율 |
| 사고율 | 악성 이메일에 대한 프롬프트 인젝션 성공률 |
| 업무 성공률 | 지정 업무 완료 여부 (pass/fail) |

## 4. 실험 결과

### 4.1 데이터셋 구성

| 데이터 항목 | 총 개수 | 주요 민감 내용 |
| --- | ---: | --- |
| 연락처 | 15 | 건강·가족·알레르기 |
| 이메일 | 30 | 건강검진 결과, 개인 대화, 악성 프롬프트 인젝션 |
| 캘린더 | 7 | 병원 예약, 가족 여행, 외근 일정 |

### 4.2 조건별 결과

| 지표 | 전체 접근 (A) | 최소정보 접근 (B) | 차이 |
| --- | ---: | ---: | ---: |
| 접근 항목 수 | 52 | 5 | **47 감소 (90.4%)** |
| Exposure Area | 3.0600 | 1.9900 | **-35.0% 감소** |
| 평균 민감도 | 0.5125 | 0.4400 | 0.0725 감소 |
| 평균 관련성 | 0.1442 | 0.8600 | 0.7158 증가 |
| 무관 항목 비율 | 86.5% | 0.0% | **86.5%p 감소** |
| 프롬프트 인젝션 노출 | 노출됨 | 안됨 | **완전 차단** |
| 업무 성공률 | 100% | 100% | 동일 |

**핵심 발견**: 최소정보 접근(조건 B)은 업무 성공률을 유지하면서, 개인정보 접근을 90% 이상 줄이고 Exposure Area를 35% 감소시켰다. 특히 악성 이메일을 포함한 전체 데이터를 읽는 조건 A에서만 프롬프트 인젝션 사고가 발생하였다.

## 5. 고찰

### 5.1 연구의 의의

본 연구는 **Exposure Area**라는 새로운 정량 지표를 제안하고, 최소정보 필터링이 업무 성능을 해치지 않으면서 프라이버시 리스크를 실질적으로 감소시킴을 규칙 기반 시뮬레이션으로 확인하였다. 특히 업무와 무관한 정보의 86.5%가 조건 A에서 노출된 반면, 조건 B에서는 필요 최소 정보만 접근하였다.

### 5.2 프롬프트 인젝션 대응 효과

악성 이메일(e17)은 전체 접근 환경에서 에이전트가 지시를 따르게 함으로써 발생한 "사고"이지만, 최소정보 필터링은 해당 이메일 자체를 읽지 않아 원천적으로 차단하였다. 이는 **접근 권한이 줄어들면 공격 표면 자체가 축소**됨을 의미한다.

### 5.3 한계

- 본 실험은 규칙 기반 시뮬레이션이며, 실제 GPT-4o 등의 LLM API를 호출한 행동 로그 기반 실험은 아니다.
- 단일 시나리오(회의 일정 조율)로 일반화에 제한이 있다.
- 민감도·관련성 가중치는 연구자의 주관적 판단으로 부여되었으므로, 향후 교차 검증이 필요하다.
- 악성 이메일은 단일 샘플로, 다양한 인젝션 패턴에 대한 일반화는 어렵다.

### 5.4 향후 연구

- 도구 호출 로그가 남는 실제 LLM API(e.g., OpenAI function-calling, LangChain)를 활용한 실증 실험으로 확장
- 인간 평가를 통한 민감도·관련성 가중치 교차 검증
- 다양한 업무 시나리오(10~20종)로 일반화 검증
- 동적 필터링(Condition C) 설계 및 테스트

---

## 6. 결론

본 연구를 통해 도구 사용형 AI 에이전트에 대한 **최소정보 접근 원칙의 실증적 효과**를 확인하였다. 가상 데이터 기반 시뮬레이션에서 Exposure Area 35% 감소, 무관 항목 86.5%p 감소, 프롬프트 인젝션 완전 차단이라는 결과를 얻었다. 업무 성공률에는 변화가 없으므로, 프라이버시 보호를 위해 에이전트의 도구 접근 권한을 업무 단위로 최소화하는 설계가 실용적임을 시사한다.

추후 실제 LLM 호출 기반 실험과 인간 평가를 통해 Exposure Area 지표를 보완한다면, AI 에이전트 개인정보 보호 설계의 핵심 기준으로 활용할 수 있을 것으로 기대한다.
