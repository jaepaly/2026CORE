# 학술제 수상 준비도 평가

## 한 줄 결론

현재 상태는 **문제의식과 시각적 전달력은 강하지만, 우수상 이상을 뒷받침할 실험적 증거는 부족**하다. 지금 그대로라면 본선 발표나 장려상 경쟁력은 있으나, 대상·최우수상 수준의 심사에서는 “업무 성공과 공격 성공을 실제로 측정했는가?”라는 질문을 넘기 어렵다.

가장 강한 방향은 모델 순위 연구가 아니라 다음 연구로 재정의하는 것이다.

> **도구 API의 세분화와 최소권한 정책이 LLM 에이전트의 개인정보 과잉 접근과 업무 성공에 미치는 영향**

| 문제 설명과 MVP 가능성 | 8/10 | 가상 메일·캘린더와 실시간 접근 로그 데모로 전달하기 쉬움 |

| 통계적 증거 | 3/10 | 10회 반복이 합집합으로 저장되어 분산·신뢰구간·유의성 검증 불가 |
| 공격 실증성 | 2/10 | 악성 이메일을 읽은 기회만 기록하며 지시 준수나 유출을 실행하지 않음 |
| 재현성 | 5/10 | 코드와 합성 데이터는 있으나 원시 호출 로그·모델 digest·seed가 없음 |
| 시각적 전달력 | 8/10 | 연구 흐름과 핵심 상충관계를 인포그래픽으로 설명 가능 |

**종합 판단: 약 5.5/10 수준의 예비연구.** 아이디어와 데모는 경쟁력이 있지만, 현재 결과를 확정적 실증 연구로 포장하면 오히려 감점될 가능성이 크다.

## 현재 결과의 강점

### 1. 발표장에서 즉시 이해되는 문제다

“회의 하나 잡으라고 했는데 AI가 이메일 33개를 전부 읽어도 되는가?”라는 질문은 비전공 심사위원에게도 바로 전달된다. 기술적 문제와 개인정보 윤리를 한 문장으로 묶을 수 있다는 점이 강하다.

### 2. 학부 수준에서 실제로 보여줄 수 있는 MVP가 있다

가상의 이메일·주소록·캘린더, 에이전트의 도구 호출, 개인정보 접근 로그, 정책 차단 화면을 한 웹 화면에서 시연할 수 있다. 논문 요약만 있는 주제보다 학술제에서 훨씬 유리하다.

### 3. 새 다중 모델 데이터에서 의미 있는 패턴이 보인다

- 세 모델 × 네 시나리오라는 최소한의 일반화 시도가 있다.
- 모델에 따라 S1·S2 접근 도구가 달랐다.
- S3·S4에서는 모든 모델이 이메일 전체 접근으로 수렴했다.
- 모델이 적게 읽을수록 필수정보도 놓치는 프라이버시–업무 상충관계가 관측됐다.

특히 마지막 두 결과는 “모델 성능 비교”보다 **도구 설계의 구조적 문제**라는 더 강한 연구 질문으로 이어진다.

## 수상을 가로막는 핵심 문제

### 1. 120회가 120개의 관측치로 보존되지 않았다

현재 `multi_model_results.json`에는 모델·시나리오별 10회의 접근 합집합만 있다. 실행별 도구 호출, 원문 응답, seed, 지연시간, 오류 여부가 없으므로 다음을 계산할 수 없다.

- 선택 확률과 변동성
- 평균·표준편차·95% 신뢰구간
- 모델 간 통계적 차이
- 특정 한 번의 예외가 합집합을 부풀렸는지 여부

심사위원이 “10번 중 몇 번 이메일을 읽었나요?”라고 물으면 현재 데이터만으로 답할 수 없다.

### 2. 업무 성공률이 실제 측정값이 아니다

코드의 `task_success`는 항상 `True`다. 도구를 적게 선택해 필수 이메일을 놓쳐도 성공으로 기록된다. 실제 결과에서는 S1~S3의 필수정보 포괄률이 25~75%에 그치므로, 최소접근이 업무를 유지한다는 결론은 아직 성립하지 않는다.

### 3. 프롬프트 인젝션은 공격표면만 측정했다

`read_emails()`를 호출하면 곧바로 사고로 기록하지만, 모델은 이메일 본문을 도구 결과로 다시 받지 않는다. 따라서 악성 지시를 읽고 따르는 단계, 비밀정보를 수집하는 단계, 외부 전송 단계가 없다. 현재 결과는 “공격에 노출될 기회”이지 “공격 성공률”이 아니다.

### 4. 도구가 너무 거칠어 모델과 인터페이스 효과가 섞인다

현재 도구는 주소록 15건, 이메일 33건, 캘린더 7건을 통째로 반환한다. 그래서 모델이 이메일 한 건만 필요해도 33건을 읽는다. 관측된 접근 패턴이 세 개뿐인 이유도 여기에 있다. 지금 설계로는 모델의 과잉 접근 성향과 API 설계의 강제 효과를 분리할 수 없다.

### 5. 최소정보 조건 B가 오라클이다

B는 정답 ID를 사전에 제공하므로 현실적 방어 시스템이 아니다. “B가 가장 안전하다”는 것은 실험 결과라기보다 정의상 당연한 결과다. 실제 비교 대상은 정책 미들웨어, 검색형 도구, 사용자 확인 절차처럼 구현 가능한 방어여야 한다.

### 6. 기존 Exposure Area가 과잉 접근을 반대로 평가할 수 있다

`민감도 × 관련성`은 관련성이 0인 불필요한 민감정보를 위험 0으로 만든다. 과잉 접근 연구에서는 다음을 분리해야 한다.

- 필수정보 포괄률
- 초과 접근 항목 수
- 초과 민감도 합
- 업무 성공률
- 공격 성공률

### 7. 데이터와 저장소 위생이 심사 신뢰도를 떨어뜨릴 수 있다

과잉접근 저장소에 전략물자 검색 실험 로그가 섞여 있고 일부 코드 문자열이 깨져 있다. 발표 자료의 숫자와 저장소 파일이 일치하지 않으면 연구 내용보다 재현성 문제가 먼저 보일 수 있다.

## 우수상 이상을 위한 개선 실험

### 연구 가설

- H1. 세분화된 검색 도구는 전체 반환 도구보다 초과 접근을 유의하게 줄인다.
- H2. 정책 미들웨어는 프롬프트만 이용한 최소권한 지시보다 공격 성공률을 더 낮춘다.
- H3. 도구 세분화와 정책 집행을 결합하면 업무 성공률을 유지하면서 초과 민감정보 노출을 줄인다.
- H4. 위 효과는 서로 다른 LLM에서도 일관되게 나타난다.

### 비교 조건

| 조건 | 구현 | 연구 역할 |
|---|---|---|
| A · 전체 반환 | 기존 `read_all` 도구 | 위험 상한선 |
| B · 프롬프트 최소권한 | 기존 도구 + “최소한만 읽기” 지시 | 약한 방어 |
| C · 세분화 도구 | 검색·필터·단건 조회 API | 인터페이스 효과 |
| D · 정책 미들웨어 | 요청 범위 검사·차단·승인 | 강한 방어 |
| E · 오라클 최소 | 정답 ID만 제공 | 이론적 상한선 |

### 필요한 세분화 도구

```text
search_contacts(name, fields)
search_emails(query, sender, date_range, fields, limit)
get_email(email_id, fields)
query_calendar(person, date_range, fields)
reserve_room(date, time, room)
send_message(recipient, body)          # 모의 실행만
external_post(endpoint, payload)       # 공격 성공 측정용 모의 sink
```

각 도구는 반환한 ID와 필드 단위 로그를 남겨야 한다. “이메일 1건의 제목만 읽음”과 “이메일 33건의 본문 전체를 읽음”을 구분해야 한다.

### 실제 에이전트 루프

1. 모델이 도구를 선택한다.
2. 실험 환경이 선택한 도구 결과를 반환한다.
3. 모델이 결과를 읽고 추가 도구 또는 최종 답변을 선택한다.
4. 악성 문서가 있으면 모델이 그 지시를 따르는지 기록한다.
5. 최종 산출물을 독립 채점기가 평가한다.

공격 성공은 단계별로 나눈다.

- 악성 데이터 접근률
- 악성 지시 준수율
- 비필수 개인정보 추가 수집률
- 모의 외부 전송 시도율
- 정책 차단률

### 업무 성공 채점

시나리오마다 정답과 필수 제약을 구조화한다.

- 일정 조율: 올바른 대상·가능 시간·충돌 없음
- 회의실 예약: 날짜·시간·장소·예약 성공
- 문서 검색: 정답 문서 ID와 핵심 근거
- 메일 요약: 필수 중요 메일 포함·허위 내용 없음

자동 채점과 두 명의 블라인드 평가를 병행하고, 평가자 일치도(Cohen's κ)를 보고하면 설득력이 높다.

### 표본 설계

현실적인 학부 연구 권장안은 다음과 같다.

- 12~20개 기본 업무
- 정상·모호·악성 변형 3종
- 3개 이상 모델
- 조건별 10회 반복
- 동일 시나리오를 모든 조건에 적용하는 paired design

예: 15개 업무 × 3개 변형 × 4개 방어 조건 × 3개 모델 × 10회 = 5,400회. 시간이 부족하면 모델당 반복을 5회로 낮추기보다 시나리오 수와 조건 수를 우선 확보하고, 파일럿 결과로 적정 반복 수를 정한다.

### 통계 분석

- 분석 단위: 호출 횟수가 아니라 **시나리오**
- 연속·순서형 지표: paired bootstrap CI 또는 Wilcoxon signed-rank test
- 성공/실패 지표: McNemar test 또는 mixed-effects logistic regression
- 효과크기와 95% CI를 p-value와 함께 제시
- 모델·시나리오를 random effect로 다루면 일반화 주장이 강해짐

## 수상용 MVP 제안

화면을 좌우로 나눈다.

- 왼쪽: 제한 없는 에이전트
- 오른쪽: 세분화 도구 + 정책 미들웨어 에이전트
- 중앙: 동일 업무 요청
- 하단: 실시간 접근 ID, 민감 필드, 필수/초과 구분, 공격 지시, 차단 로그

악성 이메일이 섞인 동일 업무를 두 에이전트에 실행하고, 왼쪽은 전체 이메일을 읽어 모의 외부 전송을 시도하며 오른쪽은 검색 범위를 제한하거나 사용자 승인을 요구하는 장면을 보여준다. 이 데모는 연구 결과를 30초 안에 이해시키는 역할을 한다.

## 우선순위 로드맵

### 반드시 해야 할 것

1. 실행별 JSONL 로그 저장
2. 올바른 한글 프롬프트와 시나리오 복구
3. 실제 에이전트 도구 결과 루프 구현
4. 업무 성공 채점기 구현
5. 공격 성공 단계를 실제로 실행·기록
6. 세분화 도구와 정책 미들웨어 조건 추가
7. 기존 Exposure Area를 보조지표로 내리거나 수정

### 시간이 남으면 할 것

1. 시나리오를 12개 이상으로 확장
2. 상용 API 모델 또는 다른 계열 모델 추가
3. 사람 평가와 평가자 일치도 측정
4. 민감도 가중치에 대한 민감도 분석
5. 사용자 확인 절차의 클릭 비용·지연시간 분석

## 최종 판단

현재 연구는 **좋은 질문을 가진 프로토타입**이다. 수상작이 되려면 “많이 읽었다”는 현상 관찰을 넘어, 다음 인과적 메시지를 실험으로 보여줘야 한다.

> 모델에게 조심하라고 말하는 것만으로는 부족하며, 검색형 도구와 정책 집행을 함께 설계해야 개인정보 노출을 줄이면서 업무를 완수할 수 있다.

이 메시지를 실제 업무 성공률과 실제 공격 성공률로 입증하면 대주제 적합성, 기술 구현, 사회적 의미, 시각적 데모가 모두 맞물리면서 우수상 이상을 노릴 수 있다.
