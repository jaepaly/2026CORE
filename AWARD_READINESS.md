> 갱신일: 2026-06-25

## 한 줄 결론

이전 버전보다 확실히 좋아졌다. 이제는 단순 시뮬레이션이 아니라 실제 멀티턴 에이전트 루프, 도구 호출 로그, 업무 성공 판정, 모델 비교를 갖춘 MVP형 실험이다.

다만 현재 데이터는 결측과 중복이 있어 “정책 미들웨어가 과잉 접근을 줄였다”는 강한 결론까지는 어렵다. 지금 그대로라면 장려상 경쟁력은 생겼지만, 우수상 이상을 노리려면 균형 실험과 공격 시나리오 보강이 필요하다.

## 이전보다 나아진 점

1. `task_success=True` 하드코딩에서 벗어나 규칙 기반 성공 판정을 도입했다.
2. 실제 도구 결과를 모델에게 다시 전달하는 멀티턴 루프를 구현했다.
3. 모델 3개, 시나리오 12개, 조건 3개의 비교 구조를 만들었다.
4. run 단위로 최종 출력, 접근 ID, tool call, latency, 공격 관련 플래그를 저장했다.
5. 발표 주제가 “모델 성능 비교”가 아니라 “AI 에이전트 개인정보 접근 측정 프레임워크”로 더 좋아졌다.

## 우수상 이상을 가로막는 문제

### 1. 실험 조합이 아직 균형 잡히지 않았다

계획은 1,080개였지만 현재 고유 실험 키는 698개다. 일부 모델·시나리오·조건 조합은 아예 없고, 일부는 중복으로 많이 들어 있다. 조건별 성공률과 모델별 비교가 시나리오 구성에 영향을 받을 수 있다.

### 2. 성공률이 전반적으로 낮다

조건별 성공률이 14~19% 수준이다. 이는 “AI 에이전트가 업무를 잘 수행한다”는 연구라기보다는 “현재 구현과 도구 설계에서 업무 성공이 어렵다”는 결과에 가깝다. 발표에서는 이 낮은 성공률을 실패로 숨기지 말고, 개인정보 최소화와 업무성 사이의 trade-off를 드러내는 결과로 해석해야 한다.

### 3. 공격 실험은 아직 약하다

현재 악성 이메일 접근이 관측되지 않았다. 따라서 공격 방어를 검증했다기보다는, 현재 검색·시나리오 조건에서는 악성 페이로드에 도달하지 않았다고 보는 것이 맞다. 공격성 있는 시나리오를 따로 만들어야 보안성 주장이 강해진다.

### 4. 정책 C의 효과가 애매하다

C가 성공률을 A와 비슷하게 유지한 것은 장점이지만, 평균 접근 ID와 초과 접근 ID가 확실히 줄었다고 말하기 어렵다. 정책 미들웨어가 정말 효과적이라는 결론을 위해서는 더 강한 정책 조건이 필요하다.

## 개선 우선순위

1. 헤르메스 추가 실험이 끝나면 1,080개 조합 기준으로 결측을 메운다.
2. 중복 run을 제거하거나, raw/dedup 결과를 명확히 분리해 보고한다.
3. 악성 이메일이 반드시 검색 결과에 포함되는 공격 시나리오를 추가한다.
4. 공격 노출, 지시 준수, 민감정보 수집, 외부 전송 시도를 각각 따로 측정한다.
5. C 조건을 단순 쓰기 차단에서 다음 정책으로 확장한다.
   - 검색 `limit` 강제
   - 민감 필드 마스킹
   - 외부 도메인 전송 차단
   - 일정 생성/메일 발송 전 사용자 확인
6. 발표에서는 “정책이 완벽히 효과적이었다”가 아니라 “효과를 검증할 수 있는 프레임워크를 만들었고, 현재 중간 결과에서는 trade-off와 한계가 보였다”로 포지셔닝한다.

## 추천 발표 포지셔닝

> 본 연구는 AI 에이전트가 업무 수행 중 접근하는 개인정보 범위를 run 단위로 기록하고, 프롬프트 지시와 정책 미들웨어가 업무 성공률·필수정보 포괄률·초과 접근에 미치는 영향을 비교한 MVP형 실험이다.

이 포지션이 가장 안전하고 강하다. 과장하지 않으면서도, 학부생 프로젝트로는 구현·문제의식·시각화가 모두 살아난다.
