# 업무를 돕는 AI는 어디까지 읽어야 하는가?

![AI 에이전트 개인정보 과잉 접근 연구](docs/figures/readme_hero.png)

도구 사용형 AI 에이전트가 업무를 수행할 때 **민감정보 노출 위험이 모델의 성향에서 오는가, 아니면 도구 인터페이스의 권한 설계에서 오는가**를 측정하는 학부 연구 프로젝트입니다. 가상의 주소록·이메일·캘린더 환경에서 세 개의 로컬 LLM을 실제 멀티턴 에이전트 루프로 돌려, 도구가 한 번에 무엇을 반환하도록 설계되었는지에 따라 개인정보 노출과 프롬프트 인젝션 위험이 어떻게 달라지는지 비교합니다.

> **핵심 주장:** AI의 민감정보 과잉노출 위험은 "모델이 알아서 많이 읽어서"가 아니라 **인터페이스 권한 설계**에서 온다. **필드 단위 최소권한 정책 하나가 민감정보 노출 용량을 93%(151.5→10.5) 줄이고, 프롬프트 인젝션 지시 전달을 5건→0건으로 구조적으로 차단**한다 — 모델 종류와 무관하게.

## 한눈에 보는 결과

![인터페이스가 위험을 결정한다](docs/figures/fig_interface_risk.png)

- **노출 용량(설계 속성, worst-case):** 필드 필터가 없는 A는 민감필드 노출 용량 **151.5**, 악성 인젝션 지시 전달 가능 **5건**. 필드 최소권한 정책 C/D는 **10.5 / 0건**.
- **실제 접근(모델 행동):** 무방어 조건(A)에서도 모델은 평균 **0.65건**(최대 6건)만 접근. 즉 이 소형 모델들은 과잉이 아니라 **과소 접근**한다.
- **결론:** 위험은 모델 행동(낮음)이 아니라 인터페이스가 허용하는 노출 용량(설계)에 잠재한다. 안전한 선택은 모델에게 "조심하라"고 부탁하는 것이 아니라 **인터페이스가 한 번에 노출할 수 있는 범위를 설계로 제한**하는 것이다.

## 연구 질문

1. 실제 에이전트 루프에서 LLM은 업무에 필요한 것보다 많은 개인정보를 읽는가?
2. "과잉 접근"은 모델의 판단 문제인가, 도구 인터페이스 설계 문제인가?
3. 필드 단위 최소권한 인터페이스는 모델과 무관하게 노출 용량을 줄이는가?
4. 같은 인터페이스 설계가 프롬프트 인젝션 위험까지 함께 낮추는가?

## 실험 개요

![연구 설계](docs/figures/readme_method.png)

### 데이터 환경

| 데이터 도구 | 반환 범위 | 항목 수 |
|---|---|---:|
| `search_contacts` / `get_contact` | 연락처 검색·상세 | 15 |
| `search_emails` / `get_email` | 이메일 검색·전문 | 33 |
| `search_calendar` / `create_event` | 일정 검색·생성 | 7 |
| `read_all` (coarse, 위험 상한선) | 전체 통째 반환 | 55 |
| **합계** | 합성 개인정보 | **55** |

연락처 `notes`에는 임신·병원 진료·알레르기·주소 변경 등 실제로 민감한 PII가, 이메일 `body` 중 5건(`e17`·`e29`·`e31`·`e32`·`e33`)에는 프롬프트 인젝션 페이로드가 들어 있습니다.

### 비교 조건

| 조건 | 인터페이스 | 정책 |
|---|---|---|
| A | 세분화 도구 전체 허용 | 필드 필터 없음 (위험 상한선) |
| B | 세분화 도구 + "최소한만 읽기" 지시 | 프롬프트 수준 방어 |
| C | 세분화 도구 + 필드 최소권한 미들웨어 | `body`·`phone`·`notes` 등 민감필드 차단 + 쓰기 차단 |
| D | 세분화 도구 + 강한 필드 최소권한 | C보다 더 제한 + `get_email` 차단 |

세 모델 `qwen3:8b`, `llama3.1:8b`, `qwen2.5:7b`로 **3모델 × 20시나리오 × 4조건 × 10seed = 2,400 runs**를 실행했습니다. 각 run은 실제 tool-calling 멀티턴 루프이며, 실행 단위로 JSONL 로깅됩니다.

## 핵심 결과

### 1. 인터페이스가 노출 용량을 결정한다 (모델 무관)

| 정책 | 노출 항목 | **민감필드 노출 용량** | **악성 인젝션 전달 가능** |
|---|---:|---:|---:|
| A (필드 필터 없음) | 55 | **151.5** | **5건** |
| C (필드 최소권한) | 55 | **10.5** | **0건** |
| D (강한 최소권한) | 55 | **10.5** | **0건** |

민감필드 노출 용량은 `민감 필드 가중치`(전화 2, notes 3, 본문 2, 일정 1.5)의 worst-case 합으로, **모델이 무엇을 하든** 정책이 허용하는 상한선입니다. 필드 단위 최소권한이 이 상한선을 93% 낮춥니다. 계산: [`interface_risk.py`](interface_risk.py).

### 2. 실제 모델은 과잉이 아니라 과소 접근한다

![모델 비교](docs/figures/fig_model_compare_v2.png)

| 조건 | 평균 접근 ID | 필수정보 포괄률 | 접근 정밀도 | 공격 노출 |
|---|---:|---:|---:|---:|
| A | 0.65 | 20.1% | 43.5% | 0% |
| B | 0.64 | 19.1% | 40.9% | 0% |
| C | 0.64 | 18.5% | 39.8% | 0% |
| D | 1.26 | 19.7% | 30.8% | 0% |

무방어 조건(A)에서도 평균 접근은 0.65건에 불과하며, `read_all`(통째 반환) 도구를 제공해도 모델은 거의 호출하지 않았습니다. v1에서 보였던 "이메일 33개를 다 읽는다"는 모델 행동이 아니라 실험 harness가 조건 A에 55개를 일괄 할당했던 **설계 산물**이었습니다.

### 3. 같은 필드 정책이 프롬프트 인젝션을 구조적으로 차단한다

악성 인젝션 지시는 이메일 `body`에 들어 있습니다. 정책 C/D의 필드 필터는 `body`를 제거하므로 **공격 지시가 모델에 애초에 전달되지 않습니다.** 전체 2,400 runs에서 `attack_compliance = 0`, `attack_leakage ≈ 0`인데, 이는 모델이 공격을 잘 막아서가 아니라 **인터페이스가 공격 표면을 설계로 제거**했기 때문입니다.

### 4. 프라이버시–업무 상충관계

![조건 효과](docs/figures/fig_condition_effect.png)

| 모델 | A | B | C | D |
|---|---:|---:|---:|---:|
| `qwen3:8b` | 31% | 28% | 19% | 13% |
| `llama3.1:8b` | 17% | 23% | 12% | 20% |
| `qwen2.5:7b` | 22% | 23% | 10% | 10% |

업무 성공률은 전반적으로 낮습니다(조건별 13~25%). 이 소형 모델들의 실패는 과잉 접근이 아니라 **과소 접근 + 환각**(필요한 이메일·일정을 안 읽고 날짜를 지어냄)에서 옵니다. 채점은 피험 모델과 분리된 LLM-as-judge(`JUDGE_MODEL`)로 수행합니다.

**통계적 유의성** (paired McNemar test, [`stats_v2.py`](stats_v2.py)).
⚠️ 현재 데이터는 seed가 모델 호출에 전달되지 않아 동일 `(model, scenario, condition)`의 10개 seed가 사실상 복제다(접근 76%·성공 75%가 seed 전부 동일). 따라서 분석 단위는 seed가 아니라 **`(model, scenario)` 페어(n=60)**이며, seed는 majority로 붕괴한다.

| 비교 | McNemar p (n=60) | 해석 |
|---|---:|---|
| A vs B | 0.69 | 프롬프트 최소화는 유의한 영향 없음 |
| A vs C | 0.23 | 방향상 낮으나 **유의하지 않음** |
| A vs D | 0.23 | 유의하지 않음 |
| C vs D | 1.00 | 차이 없음 |

성공률(collapsed): A 0.20, B 0.23, C 0.12, D 0.12.

→ **종합:** 필드 단위 최소권한은 노출 용량을 93% 줄이고 인젝션을 구조적으로 차단하지만, 그에 따른 업무 성공률 하락(0.20→0.12)은 **현재 표본에서 통계적으로 유의하지 않다(방향만 시사).** 통계 검정력을 확보하려면 seed를 실제로 모델에 전달해 독립 반복을 만들거나(코드 수정 완료, 재실행 필요) 시나리오 수를 늘려야 한다. 노출 용량·인젝션 차단 결과는 분석적·구조적 사실이므로 이 한계의 영향을 받지 않는다.

## 해석 가능한 주장과 아직 할 수 없는 주장

### 말할 수 있는 것

- 필드 단위 최소권한 인터페이스는 모델과 무관하게 민감정보 노출 용량을 93% 줄인다(151.5→10.5).
- 같은 필드 정책이 인젝션 지시 전달을 5→0건으로 구조적으로 차단한다.
- 실제 에이전트 루프에서 이 소형 모델들은 과잉이 아니라 과소 접근하며, 위험은 행동이 아니라 인터페이스 설계에 잠재한다.

### 아직 말할 수 없는 것

- 더 크고 tool-eager한 상용 모델(GPT/Claude 등)에서도 과소 접근이 유지되는지.
- 낮은 업무 성공률이 모델 능력 한계인지 채점 엄격성인지의 분리.
- 민감도 가중치 선택에 대한 민감도 분석.
- 프레이밍("안전하다/권한 있다")이 접근 범위에 미치는 영향 (v3 인프라로 측정 가능, 보조 과제).
